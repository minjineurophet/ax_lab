02

LLM의 본질

⬡ LLM은 정답 DB가 아니다
LLM은 확률로 다음 말을 예측하는 기계입니다.

예시: "내일이 다음에 뭐?" → 학습 데이터에 '네이버'가 많이 등장했다면 '네이버'가 나올 확률이 높아집니다. 지식을 꺼내는 게 아니라, 확률적으로 다음 단어를 생성하는 것입니다.
⬡ 질문 범위가 확률을 결정한다
나쁜 예: "뉴로핏은 어떤 회사야?" → 전체 확률 공간에서 탐색

좋은 예: "뉴로핏은 신경과학 및 바이오 분야에서 어떤 회사야?" → 탐색 범위가 좁아져 원하는 답의 확률이 급상승

컨텍스트 관리의 두 가지 함정

⚠ Context Rot (맥락 부패)

너무 이것저것 많은 정보를 한 번에 주면 헷갈림. 필요한 맥락만 딱 모아서 관계를 명확하게 줘야 한다. 쓰레기 맥락이 들어가면 성능이 곱해지듯 떨어진다.

⚠ Context Bloat (비대)

같은 맥락이라도 너무 길게 주면 중원보원처럼 흐려짐. 간결하고 명확하게, 불필요한 내용은 제거. 에이전트 MD도 짧고 명확하게 작성해야 성능이 좋다.

03

프롬프트 엔지니어링

좋은 프롬프트는 AI에게 신입사원에게 업무를 설명하는 것처럼 구체적으로 줘야 합니다. "내가 이렇게 이렇게 했는데, 이걸 해줘" 수준이 기본입니다.

요소 설명 예시 필수
역할 누구의 입장에서 일하는지 "너는 의료기기 인허가 전문가야" 옵션
목표/배경 왜 이 작업을 하는지 맥락 "우리 회사는 뇌 분석 소프트웨어를 만드는 뉴로핏이야" 필수
입력 어떤 데이터/파일이 들어오는지 "이 CSV 파일에는 이름, 나이, 진단명이 있어" 필수
절차 단계별로 어떻게 처리할지 "1) 먼저 헤더 확인 2) 조건 필터 3) 결과 정리" 필수
조건/제약 지켜야 할 규칙 "MVP로, 오버스펙 하지 말 것" 필수
출력 형식 결과물의 형태를 명시 "마크다운 표로, 최대 10줄" 필수

실전 팁: 5Y 질문법

◈ 5Y (Five Whys) 기법
요구사항이 막막할 때: "나한테 5가지 질문을 던져서 내 요구사항을 구체화시켜 줘"
AI가 5개의 질문을 하면서 원하는 결과물의 범위를 좁혀줍니다. 막연한 아이디어를 뾰족한 스펙으로 만드는 방법.
04

하네스(Harness) — AI 일 잘하게 만드는 환경 설계

하네스란?

말에게 마구(馬具)를 씌우듯, AI가 일을 잘할 수 있는 환경을 설계하는 것. 모델만 쓰면 "로우 모델"이고, 도구를 연결하고 규칙을 만들어 작업을 잘하게 만들면 "에이전트 하네스"입니다. 개발용 하네스(코덱스/코파일럿의 기본 설정)는 내 업무에 최적화되어 있지 않으므로, 나만의 사용자 하네스를 만들어야 합니다.

에이전트 MD
AI 에이전트의 안내판/규칙서. 어떤 파일이 어디 있는지, 어떤 방식으로 일해야 하는지, 어떤 제약이 있는지를 정의. 짧고 명확하게 작성할수록 성능이 좋다. 오픈AI 공식 가이드도 "간결하게 작성하라"고 명시.
스킬 (Skill)
업무 매뉴얼. 청소기를 샀을 때 설명서를 보면 더 잘 쓰는 것처럼, AI가 특정 작업의 매뉴얼을 보고 안 보고의 차이. 성공한 작업을 스킬로 기록해두면 다음 번에 그 수준부터 시작.
MCP 서버
AI에게 팔다리를 달아주는 것. 원래 입만 있던(답변만 하던) AI가 도구를 쓸 수 있게 되는 장치. 팀즈, 슬랙, 아마란스 등 외부 서비스와 연결. 단, 많이 연결할수록 컨텍스트를 차지하므로 필요한 것만 켜두기.
서브 에이전트
메인 에이전트가 전문 분야별 서브 에이전트에게 위임하는 구조. 핵심: 검수는 항상 맥락 없이 서브 에이전트에게 시킬 것. 자기가 만든 거를 자기가 검수하면 "잘했을 것"이라 가정하고 넘어감.
✕ 나쁜 방식

메인 에이전트가 작업하고, 같은 맥락에서 스스로 검수. "내가 잘했을 것"이라 가정해서 꼼꼼히 안 봄. 두세 번 반복 작업 발생.

✓ 좋은 방식

메인 에이전트가 작업 → 맥락 없이 서브 에이전트에게 "검수해라"만 전달 → 서브 에이전트가 편견 없이 꼼꼼하게 검수. 한 번에 끝남.

05

업무 자동화 구축 원칙

①

워크플로우를 그대로 따와야 사람들이 쓴다 (가장 중요)

구성원들이 평소에 어떻게 일하는지를 정리하고, 그 흐름을 그대로 시스템에 녹여야 합니다. "노션 어려운 사람이 많으면 노션 못 쓰는 것"과 같은 논리. 조금이라도 기존 흐름과 다르면 아무도 안 씁니다.

②

업무를 태스크 단위로 쪼개기

예시: 실험 진행 → 결과 확인 → 특정 조건에서 이상 여부 분석 → 보고서 전달
이처럼 하나의 업무를 작은 태스크로 쪼개고, 각 태스크를 컴퓨터가 처리하는 방식으로 치환해 적으면 자동화의 80%가 해결됩니다.

③

MVP로 시작, 절대 오버스펙 금지

처음부터 완벽하게 만들려 하면 AI가 AWS S3, Lambda 등 월 100만 원짜리 인프라를 자동으로 구성합니다. 프롬프트에 반드시 MVP 형태로, 오버스펙 하지 말 것을 명시할 것.

④

프런트엔드부터 만들어라 (비개발자)

눈에 보여야 이해가 됩니다. 백엔드부터 만들고 프런트엔드를 붙이는 방식은 비개발자에게 너무 어렵습니다. 시각적으로 확인하면서 단계별로 만들어가야 포기하지 않습니다.

⑤

새 세션을 자주 열 것 (컨텍스트 관리)

작업 중간중간 세션을 교체해야 쓰레기 맥락이 쌓이지 않습니다. "다른 작업을 시작할 때", "검증 후 새로운 작업에 착수할 때" 플러스 버튼으로 새 창을 열고, 필요한 맥락만 요약해서 가져가세요.

⑥

성공한 작업은 반드시 스킬로 기록하기

잘 된 작업의 프로세스를 스킬(MD 파일)로 저장해두면, 다음에 비슷한 작업을 할 때 처음부터 설명할 필요가 없습니다. 에이전트가 스킬을 참고하면 같은 수준으로 바로 시작합니다.

07

MCP — AI에게 팔다리 달아주기

MCP(Model Context Protocol)란?

원래 AI는 "입"만 있어서 답변만 했습니다. MCP는 AI에게 눈·손·발을 달아주는 외부 도구 연결 규약입니다. 파일을 읽고, 코드를 실행하고, 슬랙·팀즈·아마란스 같은 외부 서비스에 접근하는 것이 모두 MCP를 통해 이뤄집니다. "모델이 컨텍스트를 가져올 수 있는 통신 교량"이라고도 부릅니다.

✕ MCP 없는 AI
입만 있는 상태. "이렇게 하면 돼요"라고 알려줄 뿐, 실제로 파일을 열거나 버튼을 클릭하거나 서비스에 접근할 수 없습니다.
✓ MCP 있는 AI
눈(파일 읽기), 손(코드 실행·파일 쓰기), 발(브라우저·외부 서비스 접근)이 생깁니다. 에이전트가 도구를 호출해 실제로 일을 완료합니다.
⚠ 주의사항
MCP를 많이 연결할수록 컨텍스트를 차지합니다. 필요한 MCP만 켜두고, 사용하지 않는 것은 비활성화하는 습관이 중요합니다.

MCP 찾는 방법

1

Marketplace에서 검색

Claude Code → 톱니바퀴(설정) → MCP → 브라우저 마켓플레이스에서 원하는 서비스명 검색. 스타 수가 적은 것은 무시하고 공식 또는 스타 수 많은 것을 선택.

2

Skills.sh에서 스킬 형태로 가져오기

skills.sh에서 원하는 도구명 검색 → 이미 잘 정리된 MCP 활용 스킬을 복사해서 에이전트 MD나 스킬 폴더에 넣으면 즉시 활용 가능.

3

공식 GitHub 링크 + AI에게 설치 위임

공식 GitHub에서 MCP 링크 복사 → 에이전트에게 "이 링크 참조해서 MCP 설치하고 연결해줘"라고 요청. 순서 정리, 설치, 연결까지 AI가 처리. 막히면 "순서 정리해줘"로 먼저 계획 수립 후 진행.

4

보안 환경에서 막힐 때

회사 보안 정책으로 MCP 설치가 막힐 수 있음. 이 경우 서버에 포트 하나 뚫어서 연결하거나, 스킬(MD 파일)로 맥락만 제공하고 AI가 판단하게 하는 방식으로 우회. 어떤 방법이 가능한지 보안팀과 먼저 확인.

MCP 직접 만드는 법 (커스텀 MCP)

언제 만드는가?

Marketplace에 원하는 서비스가 없거나, 내부 사내 시스템(아마란스, 자체 DB 등)에 연결해야 할 때 직접 만듭니다. 구조를 이해하면 AI가 대부분 코드를 만들어줍니다.

①

MCP 서버 구조 이해하기

MCP 서버는 JSON 형태의 키-값 구조로 도구(Tool)를 정의합니다. 각 도구에는 이름, 설명(description), 파라미터가 있습니다. AI가 description을 읽고 언제 이 도구를 써야 할지 판단합니다.

②

description을 명확하게 작성하기

MCP 성능 문제의 대부분은 description이 부실해서 생깁니다. AI가 "이 도구를 언제, 어떤 입력으로, 어떤 결과를 위해 써야 하는지"를 명확히 알 수 있도록 작성. 영어로 구체적으로 적을수록 좋음.

③

에이전트에게 제작 위임하기

공식 MCP 링크(예: modelcontextprotocol.io)를 에이전트에게 주고, "이 링크 참조해서 [서비스명] MCP 서버를 만들어줘"라고 요청. 이후 설치, 테스트, 에이전트 MD 등록까지 위임.

④

로컬 vs 원격 서버

로컬: 내 컴퓨터에서 돌리는 방식. 보안에 민감한 정보(사번, 사내 DB)를 다룰 때 적합.
원격(URL 방식): 서버에 띄워두고 여러 사람이 공유해서 쓰는 방식. 팀 단위 공유 도구에 적합.

⑤

스킬로 등록해서 재사용하기

만들어서 잘 동작하는 MCP 설정을 스킬(MD 파일)로 저장. 이후 비슷한 프로젝트를 시작할 때 스킬을 불러오면 처음부터 다시 만들 필요가 없음.

MCP 서버 기본 구조 예시 (Claude Code 설정 파일)

// .claude/settings.json 에 MCP 서버를 등록하는 예시 { "mcpServers": { "slack": { "command": "npx", "args": ["-y", "@modelcontextprotocol/server-slack"], "env": { "SLACK_BOT_TOKEN": "xoxb-your-token", "SLACK_TEAM_ID": "T00000000" } }, "custom-amaranth": { "command": "node", "args": ["./mcp-servers/amaranth.js"], // 사내 시스템은 로컬 파일로 직접 구현 "env": { "API_KEY": "..." } } } }

커스텀 도구(Tool) 정의 예시 — description이 핵심

// 사내 DB에서 데이터를 가져오는 커스텀 도구 정의 { "name": "query_medical_data", "description": "의료 영상 데이터베이스에서 조건에 맞는 환자 데이터를 조회합니다. 입력: age_range(나이 범위), diagnosis(진단명), modality(촬영 방식 MRI/CT 등). 반환: 조건에 맞는 레코드 목록 (JSON). 언제 사용: 연구자가 특정 조건의 영상 데이터를 요청할 때.", "inputSchema": { "type": "object", "properties": { "age_range": { "type": "string", "description": "예: '20-40'" }, "diagnosis": { "type": "string" }, "modality": { "type": "string", "enum": ["MRI", "CT", "PET"] } }, "required": ["modality"] } }

MCP 활용 실전 사례 (강의 중 언급)

연결 대상 활용 방법 효과
Slack Codex에 Slack MCP 연결 → 채널에 "@codex 이 기능 만들어줘" 멘션만으로 PR 자동 생성 컴퓨터·핸드폰 어디서든 업무 지시 가능
MS Teams 팀즈 채널 메시지를 읽어 요약·분류·자동 응답 처리 수기 엑셀 정리 → 자동화
Google Drive 드라이브 문서 직접 읽기·쓰기. 규제 가이드라인 변경 감지 후 내부 문서에 자동 반영 문서 관리 자동화
사내 DB (커스텀) 아마란스 등 사내 시스템 API → 커스텀 MCP 서버로 래핑 → 자연어로 데이터 조회 SQL 없이 자연어 검색
PubMed (논문) 의료전화네 MCP + PubMed MCP 조합 → 논문 자동 검색 및 근거 요약 리서치 시간 대폭 단축
GitHub 이슈 생성·PR 병합·커밋 메시지 자동 작성까지 에이전트가 처리 개발 사이클 단축
✕ 흔한 실수

MCP를 무조건 많이 연결해 놓으면 좋다고 생각. 실제로는 컨텍스트를 차지해 AI 성능 저하. 사용 안 하는 MCP는 반드시 비활성화.

✓ 올바른 방식

현재 작업에 필요한 MCP만 켜두기. 서브 에이전트별로 전용 MCP만 붙이는 방식으로 컨텍스트 오염 방지. 에이전트 MD에 "어떤 MCP를 언제 쓸지" 명시.

08

스킬(Skill) — AI를 위한 업무 매뉴얼

스킬이란?

새로 산 청소기에 설명서가 있으면 기능도 다 써보고 더 잘 쓰게 되는 것처럼, AI에게 특정 작업의 매뉴얼을 주는 것이 스킬입니다. 마크다운(MD) 파일로 작성해 .claude/skills/ 폴더에 저장하면, 에이전트가 해당 작업을 할 때 자동으로 참고합니다. 스킬이 있고 없고의 차이 = 매뉴얼 보고 일하는 사람 vs 맨땅에서 시작하는 사람.

왜 스킬을 만드는가?

①

매번 같은 설명을 반복하지 않아도 된다

예: 식약처 인허가 작업을 할 때마다 "IEC 62366이 뭔지, 어떤 형식으로 문서를 써야 하는지"를 설명해야 한다면 매우 비효율적입니다. 스킬로 등록해두면 한 번만 만들고, 이후에는 불러서 쓰기만 하면 됩니다.

②

성공한 작업 수준을 재현한다

잘 된 작업의 프로세스를 스킬로 기록해두면, 다음에 비슷한 작업을 할 때 처음부터 시작하지 않고 그 수준에서 바로 시작합니다. 연구에 따르면 사용자가 직접 만든 스킬이 AI가 만든 것보다 성능이 더 좋습니다.

③

에이전트를 업무 전문가로 특화한다

재무팀 에이전트에게 아마란스 사용법 스킬을 주고, QA 에이전트에게 IEC 규격 스킬을 주면, 각 분야에 특화된 전문가 에이전트가 됩니다. 범용 AI와 결과물 품질이 달라집니다.

④

팀 전체가 같은 수준으로 일한다

스킬 파일을 팀원에게 공유하면 팀 전체가 동일한 기준과 방식으로 AI를 활용할 수 있습니다. 개인의 노하우를 조직의 자산으로 만드는 방법.

스킬 파일 구조 (MD 형식)

--- 파일 위치: .claude/skills/iec-62366-ux.md --- # IEC 62366 유저빌리티 검토 스킬 ## 목적 의료기기 소프트웨어의 유저빌리티 엔지니어링 문서를 IEC 62366-1 기준에 맞게 작성하고 검토한다. ## 작업 순서 1. 요구사항 문서에서 사용자 인터페이스 관련 항목 추출 2. Use Specification(사용 사양서) 형식으로 정리 3. 위험 관련 사용 시나리오(HURA) 식별 4. 형성 평가 / 총괄 평가 계획 수립 ## 출력 형식 - 마크다운 표로 정리 - 각 항목에 IEC 62366 조항 번호 병기 (예: 5.1.1) - 빠진 항목은 [미작성] 태그로 표시 ## 주의사항 - 최신 개정판(2023) 기준 적용 - 식약처 가이드라인과 상충 시 식약처 우선

스킬 만드는 방법

1

성공한 작업 프로세스를 AI에게 정리시킨다

작업을 마친 뒤 새 세션에서: "방금 진행한 작업 프로세스를 자세하게 정리해서 스킬 파일로 만들어줘." 처음부터 직접 쓰는 것보다 AI가 만든 초안을 수정하는 게 빠릅니다.

2

.claude/skills/ 폴더에 MD 파일로 저장

파일명은 작업 내용을 한눈에 알 수 있게 짓습니다. 예: playwright-e2e.md, amaranth-integration.md. 같은 폴더의 파일들을 에이전트가 자동으로 참고합니다.

3

에이전트 MD에 "이 스킬을 참고하라"고 명시

에이전트 MD(AGENT.md)에 "특정 작업 시 skills/ 폴더의 해당 스킬을 반드시 먼저 읽을 것"이라고 적어두면, 에이전트가 자동으로 해당 스킬을 불러서 참고합니다.

4

스킬을 계속 개선한다

작업을 반복하면서 "이 부분은 빠졌네", "이 조건을 추가해야겠다"는 것들을 누적합니다. 스킬을 정교하게 만들수록 에이전트 결과물의 수준이 함께 올라갑니다.

📦 다른 사람이 만든 스킬 가져다 쓰기

Skills.sh(skills.sh)에 40만 명 이상이 공유한 스킬이 있습니다. 처음부터 만들 필요 없이, 검색해서 가져오면 됩니다.

①

Skills.sh에서 원하는 스킬 검색

skills.sh에 접속해서 원하는 도구나 작업명을 검색합니다. 예: "playwright", "nextjs", "excel", "figma" 등. 스타 수와 업데이트 날짜를 보고 신뢰할 수 있는 스킬을 선택.

②

스킬 링크를 AI에게 주고 설치 위임

스킬 페이지의 URL을 복사해서 에이전트에게 아래 패턴으로 요청합니다.

-- Skills.sh 스킬 설치 프롬프트 패턴 -- 주어진 링크의 스킬을 확인해서 .claude/skills/ 폴더에 설치해줘. 설치 후 해당 스킬이 제대로 동작하는지 검증까지 진행해야 한다. 링크: https://skills.sh/playwright → 에이전트가 링크 접속 → 스킬 내용 파악 → 폴더에 저장 → 동작 검증

💬 실제 강의에서 나온 스킬 설치 프롬프트 예시

-- github.com/fivetaku/gptaku_plugins 스킬 설치 예시 -- 주어진 Claude Code용 Plugin을 Copilot 너한테 설치해야 한다. 구조를 파악해서 마이그레이션 할 수 있도록 하고, 스텝바이스텝으로 설치 진행한다. 설치 후 검증까지 진행해야 한다. 핵심 포인트: ① 구조 파악 먼저 → ② 스텝바이스텝 설치 → ③ 검증 "설치해줘" 한 마디보다 단계를 명시하면 실패율이 크게 낮아진다.
③

내 업무에 맞게 커스텀 수정

가져온 스킬은 일반적인 내용으로 작성돼 있습니다. "이 스킬을 우리 회사 [업무명] 기준으로 수정해줘"라고 하면 우리 환경에 맞게 조정해줍니다. 특히 출력 형식, 주의사항, 회사 고유 규칙을 추가하면 효과가 큽니다.

④

에이전트 MD에 스킬 참조 등록

AGENT.md에 "[작업명] 작업 시 skills/[파일명].md를 반드시 먼저 읽을 것"을 추가합니다. 이렇게 하면 매번 스킬을 지정하지 않아도 자동으로 불러서 씁니다.

스킬 vs 에이전트 MD — 뭐가 다른가?

📋 에이전트 MD (AGENT.md)

에이전트 전체의 행동 규칙. "너는 어떻게 일해야 해, 어떤 파일이 어디 있어, 어떤 제약이 있어"를 정의. 프로젝트 전체에 항상 적용됨. 짧고 명확하게, 불필요한 내용 없이 작성.

🔧 스킬 (skills/*.md)

특정 작업의 전문 매뉴얼. "이 작업을 할 때는 이렇게 해라"를 정의. 필요할 때만 불러서 씀. 여러 개를 만들어두고 작업에 맞는 스킬만 활성화. 상세하고 구체적으로 작성해도 됨.

09

Goal — 완료 기준을 AI에게 명확히 주기

Goal(목표 조건)이란?

AI에게 작업을 시킬 때 "언제 이 작업이 완료된 것인가"에 대한 명확한 기준을 주는 것입니다. Goal이 없으면 AI는 스스로 "다 한 것 같다"고 판단하고 멈추는데, 실제로는 빠진 것이 있거나 검증이 안 된 상태일 수 있습니다. Goal = 완료 판정 기준.

왜 Goal을 명시해야 하는가?

①

AI는 "완료했다"고 거짓말할 수 있다

AI는 작업을 마치면 "완료했습니다"라고 하지만, 실제로는 검증을 안 했거나, 일부만 완성된 경우가 많습니다. Goal에 "테스트까지 통과해야 완료", "브라우저에서 직접 확인해야 완료"라고 명시하면 진짜 완료 기준이 생깁니다.

②

작업 범위의 경계를 만든다

Goal 없이 "이 기능 만들어줘"라고 하면 AI가 관련 없는 코드까지 건드리거나, 과도하게 오버스펙으로 만드는 경우가 생깁니다. Goal에 "이 파일만 수정", "이 기능만 추가"라고 범위를 명시하면 그 안에서만 일합니다.

③

Claude Code의 Kool 기능과 연결

Claude Code에는 Experimental 기능 중 "Kool(꼴)" 기능이 있습니다. 작업이 완료됐는지를 판단하는 이밸류에이터(검증기)가 붙어서, 될 때까지 반복 실행합니다. 이 기능을 쓸 때 Goal이 명확해야 검증기가 올바르게 판단할 수 있습니다.

④

장시간 자동화 작업의 품질을 보장한다

12시간, 이틀씩 돌리는 대형 작업에서 Goal 없이 돌리면 중간에 방향이 틀려도 모릅니다. Goal이 있으면 각 단계마다 목표 달성 여부를 자체 점검하면서 진행합니다.

Goal 없는 요청 vs Goal 있는 요청 비교

✕ Goal 없는 요청

"로그인 페이지 만들어줘."

→ AI가 코드 작성 후 "완료했습니다" 반환. 실제로 브라우저에서 열어보면 에러, 스타일 깨짐, 버튼 클릭 안 됨. 사람이 직접 확인해야 함.

✓ Goal 있는 요청

"로그인 페이지 만들어줘. 완료 조건: ① 브라우저에서 직접 열어서 확인 ② 버튼 클릭 시 오류 없음 ③ Playwright 테스트 통과해야 완료."

→ 세 조건을 모두 충족할 때까지 자동으로 수정·재시도.

Goal 작성 방법 및 패턴

완료 조건형
특정 상태가 됐을 때 완료.

"모든 페이지가 브라우저에서 정상 열리고, 콘솔 에러가 0개일 때 완료."
테스트 통과형
테스트가 통과해야 완료.

"Playwright E2E 테스트가 전부 pass할 때 완료."
체크리스트형
항목을 모두 달성해야 완료.

"① 기능 구현 ② 스타일 적용 ③ 에러 없음 ④ 커밋까지 완료."
범위 제한형
작업 범위를 명확히 제한.

"login.tsx 파일만 수정. 다른 파일 건드리지 말 것."

Goal을 포함한 실전 프롬프트 패턴

-- Goal을 명시한 완성형 프롬프트 -- 대시보드 페이지에 차트 컴포넌트를 추가해줘. 작업 범위: - dashboard.tsx 파일만 수정 - 기존 레이아웃은 변경하지 말 것 완료 조건 (Goal): 1. 브라우저에서 직접 열어서 차트가 정상 렌더링되는지 확인 2. 콘솔 에러 0개 3. 모바일 뷰(375px)에서도 깨지지 않음 4. Playwright로 클릭 테스트 통과 → 4가지 조건을 모두 충족할 때까지 수정·재시도

Claude Code Kool(꼴) 기능과 Goal

✦ Kool 기능 — 될 때까지 자동 반복
Claude Code의 Experimental(실험적) 기능 중 하나. 활성화하면 이밸류에이터(완료 판정기)가 붙어서 Goal 조건을 만족할 때까지 작업을 자동으로 반복합니다.

동작 방식:
작업 실행 → 완료 조건 점검 → 미충족 시 자동 재시도 → 조건 충족 시 종료

주의: Goal이 명확하지 않으면 이밸류에이터가 엉뚱하게 판단합니다. Goal을 구체적인 검증 가능한 조건으로 작성해야 Kool 기능이 제대로 동작합니다.

Settings → Experimental → Kool 활성화

Plan 모드 — 먼저 계획하고 실행하기

Plan 모드란?

에이전트에게 작업을 시킬 때 바로 코드를 짜는 대신, 먼저 계획(Plan)을 작성하고 확인받은 뒤 실행하게 하는 방식입니다. 사람도 무작정 일하는 것보다 계획을 먼저 세우고 하면 더 꼼꼼하게 잘하는 것과 같은 원리입니다.

왜 Plan 모드를 쓰는가?

①

복잡한 작업에서 방향이 틀리는 걸 막는다

AI가 바로 작업에 들어가면 "10번 시도하고 나서야 방향이 틀렸다"는 걸 알게 되는 경우가 생깁니다. Plan을 먼저 받아보면 내가 원하는 방향인지 1분 만에 확인할 수 있습니다. 방향이 틀렸으면 수정하고 실행하면 됩니다.

②

여러 서브 에이전트를 효율적으로 조율한다

서브 에이전트 세 개를 병렬로 돌릴 때, 각자 어떤 일을 어떤 순서로 할지 미리 계획이 있어야 충돌 없이 동작합니다. Plan 모드는 이 분업 계획서 역할을 합니다.

③

기존 코드를 건드리지 않도록 범위를 잡는다

AI가 작업하다가 관련 없는 파일을 건드리거나, 이미 잘 돌아가는 기능을 망가뜨리는 경우가 있습니다. Plan에 "기존 페이지의 코드를 유지한다"는 제약을 명시하면 범위를 지켜서 작업합니다.

④

토큰·시간 낭비를 줄인다

잘못된 방향으로 긴 작업을 완료한 뒤 처음부터 다시 하는 것보다, Plan 단계에서 방향을 잡는 게 훨씬 경제적입니다. 특히 장시간(수 시간~이틀) 돌리는 대형 작업일수록 Plan이 필수입니다.

Plan 모드 vs 즉시 실행 비교

✕ 즉시 실행 (단순 요청)

AI가 바로 작업 시작 → 중간에 방향이 틀렸다는 걸 발견 → 롤백 후 다시 시작. 짧은 작업에는 괜찮지만 복잡한 멀티 파일·멀티 에이전트 작업에서는 반복 실패 발생.

✓ Plan 모드 (계획 후 실행)

AI가 먼저 MD 파일로 계획 작성 → 내가 검토·수정 → 확인 후 실행. 방향이 처음부터 맞으므로 한 번에 완성될 확률이 높아짐. 특히 복잡한 기능·멀티 에이전트 작업에 효과적.

Plan 모드 동작 방식

1

프롬프트에 /plan 또는 "계획 먼저 세워줘" 추가

Claude Code에서는 /plan 슬래시 명령, Codex에서는 "plan mode"로 동작. 또는 프롬프트에 자연어로 "작업 시작 전 계획을 MD 파일로 먼저 작성해라"고 명시해도 됩니다.

2

AI가 PLAN.md (또는 TODO.md) 파일 작성

현재 코드베이스를 파악하고, 작업을 단계별 체크리스트로 정리합니다. 어떤 파일을 어떤 순서로 만들고, 어떤 함수를 어떻게 바꿀지가 구체적으로 나옵니다. 이 파일을 보고 내가 원하는 방향과 맞는지 확인.

3

계획 검토 및 수정

오버스펙이면 "더 미니멀하게", 빠진 게 있으면 추가, 범위가 벗어나면 제한. 이 단계에서 수정하는 게 작업 완료 후 수정하는 것보다 훨씬 빠릅니다. 계획이 마음에 들 때만 실행을 승인.

4

계획대로 실행 → 체크리스트 완료 확인

AI가 PLAN.md의 항목을 하나씩 완료 표시(✅)하며 진행합니다. 작업 중간에 계획에서 벗어나지 않는지 에이전트가 스스로 체크합니다. "현재 진행 중인 플랜을 확인해라"고 중간 점검도 가능.

5

완료 후 검증 (서브 에이전트에게 위임)

PLAN.md의 모든 항목이 완료되면, 맥락 없는 서브 에이전트에게 검수를 시킵니다. "계획대로 다 됐는지 확인해라"가 아니라 "이 코드가 제대로 동작하는지 처음부터 검증해라"로 요청.

실전 Plan 모드 프롬프트 예시

-- 복잡한 작업에 쓰는 Plan 프롬프트 패턴 -- 서브 에이전트 3개를 사용해서 다음 페이지들을 완성해줘. 작업 시작 전 PLAN.md를 먼저 작성하고, 각 에이전트가 어떤 파일을 담당하는지 명확히 분리해줘. 조건: - 기존 페이지의 코드는 절대 수정하지 말 것 - MVP 형태로, 오버스펙 하지 말 것 - 각 작업 완료 시 PLAN.md에 체크 표시할 것 → AI가 먼저 계획을 보여주면, 확인 후 "실행해"라고 승인

Plan 모드가 특히 유용한 상황

📋 멀티 페이지 작업
5개 이상의 페이지를 한 번에 만들 때. 각 페이지 순서와 공통 컴포넌트 계획을 미리 잡아야 중복 작업이 없음.
🔗 기존 코드에 기능 추가
이미 돌아가는 시스템에 새 기능을 붙일 때. 어디를 건드리고 어디는 손대지 않을지 계획으로 확인.
⚙️ 서브 에이전트 병렬 작업
에이전트 여러 개를 동시에 돌릴 때. 역할 분담이 계획에 명시돼야 서로 충돌하지 않음.
⏱ 장시간 자동화 작업
수 시간~이틀 돌리는 작업. 중간에 방향이 틀리면 모든 시간이 낭비되므로 Plan으로 방향 확정 필수.

Playwright — AI가 직접 브라우저를 테스트하게 하기

Playwright란?

Microsoft가 만든 브라우저 자동화 도구입니다. 사람이 클릭하고, 스크롤하고, 입력하는 행동을 코드로 재현합니다. AI와 함께 쓰면 "내가 만든 웹 페이지를 AI가 직접 열어서 처음부터 끝까지 테스트"하는 자동 QA 시스템이 됩니다. 맥·윈도우·리눅스 모두 지원, 오픈소스.

왜 Playwright를 쓰는가?

①

"만들었어" ≠ "동작해" — 검증이 필요하다

AI가 코드를 완성했다고 해도 실제로 버튼이 눌리는지, 페이지가 열리는지, 데이터가 제대로 나오는지는 별개입니다. Playwright가 실제 브라우저를 열어서 사람처럼 클릭·입력·확인을 해주므로, 사람이 하나하나 확인하는 수고를 없앱니다.

②

빠진 페이지나 깨진 링크를 자동으로 찾는다

만든 앱에서 "클릭해도 반응 없는 버튼", "404 뜨는 링크", "빠진 페이지"를 사람이 직접 찾으면 시간이 오래 걸립니다. Playwright는 사이트 전체를 순식간에 순회하며 문제를 찾아줍니다.

③

페이지 속도·성능 문제까지 잡는다

네트워크 요청을 캡처해서 어떤 리소스가 느린지, 이미지가 너무 큰지, 불필요한 API 요청이 많은지를 분석합니다. "배경 이미지가 너무 커서 로딩이 느리다" 같은 인사이트를 자동으로 줍니다.

④

E2E 테스트 — 시작부터 끝까지 검증

E2E(End-to-End)는 사용자가 앱을 처음 열어서 최종 결과를 얻기까지 전 과정을 자동으로 테스트하는 것입니다. 로그인 → 데이터 조회 → 저장 → 결과 확인까지 전체 흐름을 한 번에 검증합니다.

Playwright로 할 수 있는 것들

🖱 브라우저 조작 자동화
버튼 클릭, 폼 입력, 스크롤, 드래그앤드롭, 팝업 처리. 사람이 하는 모든 마우스·키보드 동작을 코드로 재현.
📸 스크린샷 자동 캡처
각 페이지·각 단계별 스크린샷을 자동으로 찍어서 저장. AI가 이미지를 보고 UI가 의도대로 렌더링됐는지 판단.
🌐 네트워크 요청 분석
페이지 로딩 중 발생하는 모든 HTTP 요청을 캡처. 느린 API, 불필요한 요청, 큰 이미지 파일 등 성능 문제 감지.
⚠ 에러 감지 자동화
콘솔 에러, JS 에러, 404 링크를 자동으로 수집. 사람이 수동으로 찾는 것보다 훨씬 빠르고 빠짐없이 감지.
🔄 반복 테스트 자동화
코드를 수정할 때마다 같은 테스트를 반복 실행. "이전에 잘 되던 기능이 새 코드로 망가졌는지" 자동으로 검증 (회귀 테스트).
✅ 크롤링 및 데이터 수집
로그인이 필요한 페이지, 동적으로 로드되는 콘텐츠도 접근 가능. Firecrawl이 막히는 곳에서도 동작.

설치 및 AI와 연동하는 방법

1

Playwright 설치

터미널(VS Code 내 터미널)에서 실행. AI에게 "Playwright 설치하고 스킬 참조해서 현재 앱 E2E 테스트 진행해줘"라고 하면 설치부터 테스트까지 알아서 처리합니다.

2

Skills.sh에서 Playwright 스킬 가져오기

skills.sh에서 "playwright" 검색 → 스킬 파일을 .claude/skills/ 폴더에 저장. 에이전트가 이 스킬을 참고하면 Playwright를 더 정확하게 활용합니다. 공식 Playwright 문서 링크를 줘도 됩니다.

3

AI에게 E2E 테스트 위임하기

새 세션을 열고 아래 패턴으로 요청합니다. "직접 실행"이 중요합니다 — 코드만 생성하고 실행을 안 하는 경우가 있으므로 명시해야 합니다.

4

결과 확인 및 자동 수정

테스트 결과(에러 목록, 스크린샷, 성능 리포트)를 AI가 분석하고 바로 수정합니다. 수정 후 다시 테스트를 돌려서 문제가 해결됐는지 확인. 이 사이클을 반복합니다.

Playwright E2E 테스트 프롬프트 패턴

-- 기본 E2E 테스트 요청 패턴 -- 주어진 링크(skills.sh/playwright)에서 스킬을 확인해서 Playwright를 설치하고 현재 앱에 대해 E2E 테스트를 진행해줘. 직접 실행해서 결과까지 확인할 것. 테스트 항목: - 모든 페이지가 정상적으로 열리는지 - 빠진 페이지나 깨진 링크가 있는지 - 기존 페이지의 기능이 정상 동작하는지 → 테스트 완료 후 문제 목록 리포트 + 자동 수정
-- 성능 분석 추가 요청 패턴 -- Playwright로 네트워크 요청을 캡처해서 로딩 속도가 느린 원인을 분석해줘. 분석 항목: - 3초 이상 걸리는 요청 - 1MB 이상의 이미지·파일 - 불필요하게 중복되는 API 요청 → 원인 파악 후 최적화 제안까지

Headless vs Headed 모드

Headless 모드
브라우저 창을 화면에 띄우지 않고 백그라운드에서 실행. 서버 환경, 자동화 파이프라인에 적합. 빠르고 리소스 적게 사용. AI가 기본으로 사용하는 방식.

빠름 리소스 적음 자동화에 적합
Headed 모드
실제 브라우저 창을 화면에 띄워서 눈으로 보면서 실행. 테스트가 어떻게 동작하는지 직접 확인할 때 사용. 디버깅, 처음 테스트 확인 시 유용.

시각 확인 가능 디버깅에 적합
💡 실전 팁 — "직접 실행"을 명시하라
AI에게 Playwright를 쓰라고 하면 테스트 코드만 생성하고 실행하지 않는 경우가 있습니다. 반드시 프롬프트에 "직접 실행해서 결과까지 확인할 것"을 넣어야 합니다. AI는 할 수 있는 것도 명시적으로 시키지 않으면 안 합니다. 브라우저 검증, 테스트 실행, 배포 모두 마찬가지입니다.

디버깅 방법론

이 프롬프트로 바이브 코딩(vibe coding) 에러의 80%를 해결할 수 있습니다. 핵심은 AI에게 과학적 방법론(가설→검증→확정)을 강제하는 것입니다.

01

콘솔 로그로 오류 확인

VS Code 에서 체크박스로 "Add console logs" 추가. 개발자 도구 콘솔을 직접 캡처하거나, 에이전트가 자동으로 콘솔 메시지를 읽게 함.

02

원인 발생점 좁히기

에러가 어느 코드 구간에서 발생하는지를 특정. 광범위한 스택 트레이스보다 실제 문제 구간을 좁혀야 합니다.

03

원인 가설 3가지 작성

예: ① 특정 라이브러리 버전 충돌 ② 비동기 처리 타이밍 문제 ③ 데이터 형식 불일치. 여러 가설을 명시적으로 세우게 합니다.

04

가설별 실패 테스트 (Failing Test)

각 가설에 대해 "이 가설이 맞다면 이런 결과가 나와야 한다"는 테스트를 만들어 재현 여부 확인. 재현되면 그게 원인.

05

확정 원인 수정

확정된 원인에 대해서만 수정. 중요: 가설이 틀렸다면 체크포인트로 롤백하고 "이 가설은 틀렸다"는 정보를 주고 새 세션에서 재시작. 틀린 가설의 맥락이 누적되면 더 이상한 방향으로 흘러갑니다.

⚠ 롤백(체크포인트)을 적극 활용하라
GitHub의 커밋 기록 = 세이브 포인트. 잘못된 방향으로 10번 시도하는 것보다 올바른 체크포인트로 돌아가서 다시 시작하는 게 훨씬 빠릅니다. VS Code Git에서 이전 커밋으로 리스토어하거나, Claude Code에서 체크포인트 기능을 활용하세요.

핵심 메시지

🔍 "검증을 무조건 시켜라"
AI는 자기가 잘했다고 거짓말하는 전제로 접근. 모든 작업 완료 후 서브 에이전트에게 맥락 없이 검수를 시키는 것이 기본.
📖 "스킬은 업무 매뉴얼이다"
새로 입사한 직원에게 매뉴얼을 주면 더 잘하듯, AI도 스킬(설명서)이 있으면 없을 때보다 훨씬 잘한다. 성공한 작업은 반드시 스킬로 기록.
🎯 "MVP로 시작하라"
처음부터 완벽한 시스템을 만들려 하면 AI가 오버스펙을 짜고, 비용이 폭발하고, 구성원이 안 씁니다. 최소 기능부터 만들어 사용성 확인 후 확장.
🔧 "환경을 설계하라"
모델을 그냥 쓰는 게 아니라, 모델이 일을 잘할 수 있는 환경(하네스)을 만드는 것이 핵심. 에이전트 MD, 스킬, MCP를 조합해 나만의 하네스를 구축.
🔄 "워크플로우를 따와라"
기존에 사람이 하던 업무 흐름을 그대로 시스템에 녹여야 구성원이 사용합니다. 조금이라도 어렵거나 불편하면 아무도 안 쓰게 됩니다.
🚀 "직접 실행하라고 시켜라"
"이렇게 하면 돼요"라고 알려주는 게 아니라 "직접 실행해서 검증까지 해라"고 시켜야 합니다. AI가 할 수 있는 작업은 반드시 실행까지 맡길 것.
📦 "성공한 것을 기록하라"
성공한 작업 프로세스를 스킬로 저장하면 다음 번에 그 수준에서 바로 시작합니다. 사용자가 직접 만든 스킬이 AI가 만든 것보다 성능이 더 좋다는 연구 결과도 있음.
