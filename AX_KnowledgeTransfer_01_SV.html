관점 전환
챗봇에서 에이전트 하네스로

질문하는 단계를 넘어, 환경을 설계하는 단계로

단순히 ChatGPT에 질문하는 것에서, 도구를 호출해 실제 업무를 수행하는 환경(하네스)을 만드는 것으로. 여러분은 이미 마지막 단계를 원하고 있습니다.

1단계 · 챗봇

질문 → 답변

2단계 · 프롬프트

질문을 잘 던지기

3단계 · 컨텍스트

맥락을 관리하기

4단계 · 에이전트 하네스 — 환경을 설계하기. 말에게 마구(馬具)를 씌우듯, AI가 일을 잘하도록 도구·규칙·매뉴얼을 갖춰주는 것. 모델만 쓰면 "로우 모델", 환경을 갖추면 "에이전트".

★ 개발용 기본 하네스(Codex·Copilot 기본값)는 내 업무에 최적화돼 있지 않습니다 → 나만의 사용자 하네스가 필요합니다.

하네스 4요소
AI를 일 잘하게 만드는 네 가지 장치

하네스를 구성하는 4가지 요소

에이전트 MD · 안내판/규칙서

어떤 파일이 어디 있고, 어떻게 일해야 하며, 어떤 제약이 있는지 정의. 짧고 명확할수록 성능이 좋습니다.

스킬 · 업무 매뉴얼

특정 작업의 설명서. 성공한 작업을 스킬로 기록해두면 다음엔 그 수준부터 시작합니다.

MCP · 팔다리

답변만 하던 AI가 외부 도구(Teams·Slack·DevOps)에 접근하게 하는 연결 장치.

서브에이전트 · 위임

전문 작업을 분리해 위임. 검수는 항상 맥락 없이 서브에이전트에게.

02
하네스 4요소
여러분의 아이디어 → 어떤 요소로 만드는가

5개 아이디어를 하네스로 매핑하면

① Bug 업무 자동화→DevOps MCP + 반복작업 스킬로 기록
② Testcase 등록→Figma MCP × DevOps MCP 체이닝, 작성→등록 위임
③ Version History→Teams·DevOps MCP + 워크플로우 자동화 스킬
④ Bug 원인 분석→서브에이전트(맥락 없는 탐색) + LiteLLM
⑤ 수정 검증 가이드→서브에이전트가 영향범위·검증포인트 분석

→ 다섯 중 셋이 MCP 연결, 둘이 서브에이전트. 그래서 다음 챕터는 MCP에 무게를 둡니다.

02
하네스 4요소
검수는 왜 맥락 없이 시켜야 하는가

④⑤ 아이디어의 핵심: 검수 패턴

Bug 원인 분석·검증 가이드처럼 "결과를 믿을 수 있어야" 하는 작업일수록, 자기가 만든 걸 자기가 검수하면 안 됩니다.

✕ 나쁜 방식

메인 에이전트가 작업하고 같은 맥락에서 스스로 검수. "내가 잘했을 것"이라 가정해 꼼꼼히 안 봄 → 재오픈·반복작업 발생.

✓ 좋은 방식

작업 후 맥락 없이 서브에이전트에게 "검수해라"만 전달. 편견 없이 꼼꼼하게 본다 → 한 번에 끝.

사이드 이펙트·재오픈 방지가 목표인 ⑤번 아이디어에 그대로 적용됩니다.

03
MCP 심화
MCP를 찾고 연결하는 4가지 방법

DevOps·Figma MCP, 어떻게 붙이나

  1. Marketplace에서 검색 — 설정 → MCP → 브라우저 마켓플레이스. 스타 적은 건 무시, 공식/스타 많은 것 선택.
  2. Skills.sh에서 스킬로 가져오기 — 잘 정리된 MCP 활용 스킬을 복사해 에이전트 MD/스킬 폴더에 넣으면 즉시 활용.
  3. 공식 GitHub + AI에게 설치 위임 — "이 링크 참조해서 MCP 설치·연결해줘". 막히면 "순서 정리해줘"로 계획부터.
  4. 보안 환경에서 막힐 때 — 서버에 포트 하나 뚫어 연결하거나, 스킬(MD)로 맥락만 주고 AI가 판단. 먼저 보안팀과 가능한 방법 확인.

MCP 심화
성능을 좌우하는 단 하나 · 로컬 vs 원격

description이 8할, 그리고 연결 방식

MCP 성능 문제의 대부분은 description이 부실해서 생깁니다. "이 도구를 언제, 어떤 입력으로, 어떤 결과를 위해 쓰는지" 영어로 구체적으로.

로컬 서버

내 컴퓨터에서 실행. 사번·사내 DB 등 보안 민감 정보를 다룰 때 적합.

원격(URL) 서버

서버에 띄워 여러 사람이 공유. ③ Version History의 PAT 토큰 + URL 방식이 여기 해당 — 팀 공유 도구에 적합.

→ ③번 아이디어는 원격+PAT 구조, ①②는 사내 DevOps 접근이라 로컬/포트 우회를 검토하세요.

구축 원칙
사람들이 실제로 쓰게 만드는 법

자동화를 성공시키는 5가지 원칙

가장 중요

워크플로우 그대로

기존 업무 흐름을 그대로 시스템에 녹여야 사람들이 씁니다. 조금만 달라도 안 씀.

분해

태스크 단위로 쪼개기

업무를 작은 태스크로 나눠 컴퓨터가 처리하도록 치환 → 80% 해결.

MVP

오버스펙 금지

"MVP로, 오버스펙 하지 말 것" 명시. 안 그러면 월 100만원 인프라를 짜버림.

시각

프런트부터

눈에 보여야 이해됨. 백엔드부터는 비개발자에게 너무 어려움.

컨텍스트

새 세션 자주

작업 전환마다 새 창. 쓰레기 맥락이 쌓이지 않게.

자산화

성공은 스킬로

잘 된 작업을 MD로 저장 → 다음엔 처음부터 설명 불필요.

04
디버깅 방법론
④ Bug 원인 분석 아이디어에 직결

로그·API 응답에서 원인 추적하기

여러분의 ④번 아이디어 — "로그/API 응답 키워드 → LiteLLM → clone된 코드에서 원인 탐색" — 은 이렇게 하네스로 풀립니다.

  1. 입력 정규화 — 로그·응답값에서 핵심 키워드를 추출하는 스킬을 먼저 만들어 둡니다.
  2. 코드 베이스 연결 — clone된 코드를 읽을 수 있게 MCP(파일 읽기)와 연결.
  3. 맥락 없는 탐색 — 서브에이전트에게 "이 증상의 원인 후보를 찾아라"만 위임 → 편향 없는 후보 도출.

Bug 추적 시간 단축이라는 목표가, 추출 스킬 + MCP + 서브에이전트 조합으로 그대로 구현됩니다.

우리팀 실사례
같은 회사에서 이미 만들어 쓰는 도구들

시연 ① docs-compare · ② CRS 리뷰보드

① docs-compare (Docs Compare)

인허가 문서의 버전 비교를 자동화한 도구. Markdown·Word 지원, diff·앵커 네비게이션·키워드 검색·편집 모드·Word export. 반복적인 버전 대조 업무를 그대로 워크플로우로 옮긴 결과물.

② CRS Contents Review Board

standalone HTML 한 파일로 동작하는 콘텐츠 리뷰보드. 오버스펙 없이, 딱 필요한 만큼만 만든 MVP의 좋은 예 — 인프라 없이도 충분히 쓸 수 있음.

시연 포인트: ①은 "워크플로우 그대로", ②는 "MVP·프런트부터" 원칙이 실제로 어떻게 구현됐는지.
